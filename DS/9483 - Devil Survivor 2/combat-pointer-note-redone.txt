[32-bit Pointer] Pointer to current combat information
+0x00 Some state variable [8-bit]
0x01 = Can perform an action
0x02 = Dialogue being displayed
+0x10 Time on current turn, in frames [32-bit]
+0x1c Could indicate whether in combat [8-bit]
bit0=A unit is moving around
bit1=Unit is hostile
+0x24 Current unit state [8-bit]
0x00 = Turn starting
0x01 = Enemy moving
0x02 = Player friendly unit is looking at menu
0x03 = Moving
+0x28 Total mission timer, in frames [32-bit]
+0x2c x-index highlighted on grid [8-bit]
+0x2d y-index highlighted on grid [8-bit]
+0x218 Actions performed this turn [Bitflags]
Some of these are duplicate. Probably helps determine priority for next turn.
bit0=Move
bit1=Attack
bit4=Move
+0x24c Current unit taking turn [8-bit]
0xff = Null (seen when mission has not started)
0xff -> 0x00 = Combat has started and protagonist is taking turn
From there, it's the team index seen in the 0th position of the turn order array.
+0x24d Number of tiles moved [8-bit]
This is misleading because it goes up even if you move then cancel, it doesn't go back down while you're on this turn.
+0x250 Turn order array [12 bytes]
This specifically holds indexes of teams. First index is active team.
Player controllable teams are loaded first and get indexes 0x00 -> 0x03. Any index not used is filled with 0xff.
+0x25c Projected turn order array [12 bytes]
Really it's a copy of the above array and I haven't figured out why yet. Occasionally there's a minor difference, so I presume it's projecting what next turn could look like.
+0x268 Correlated with suspend, but not quite the same [8-bit]
0x00 = Playing as usual
0x01 = Loaded from suspend. Important note: Dubhe also triggered this naturally
